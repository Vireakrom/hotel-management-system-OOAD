🎯 Design Patterns Implementation

This system implements 7+ design patterns to demonstrate OOAD principles:

1. Singleton Pattern

Used in: DatabaseManager class

Purpose: Ensures only one database connection instance exists throughout the application, managing connection pooling efficiently.

2. Factory Pattern

Used in: RoomFactory class

Purpose: Creates different room types (Single, Double, Suite, Deluxe) without exposing instantiation logic to the client.

3. Strategy Pattern

Used in: Payment processing (IPaymentStrategy interface)

Purpose: Allows interchangeable payment methods (Cash, Credit Card, Online Transfer) at runtime.

4. Observer Pattern

Used in: Room status notifications

Purpose: Automatically notifies housekeeping staff when room status changes (e.g., checkout triggers cleaning task).

5. Decorator Pattern

Used in: Adding additional services to bookings

Purpose: Dynamically adds extra services (breakfast, spa, airport transfer) to bookings without modifying base booking class.

6. Repository Pattern

Used in: Data access layer (BookingRepository, RoomRepository, GuestRepository)

Purpose: Abstracts data access logic, making code more testable and maintainable.

7. Facade Pattern

Used in: BookingFacade class

Purpose: Simplifies complex booking operations by providing a unified interface to multiple subsystems.
